# 모바일앱 설계 — React Native (English For Us)

> **버전** 1.0 · **작성일** 2026-06-13
> 웹과 동일한 FastAPI 백엔드를 공유하는 iOS·Android 크로스플랫폼 앱.

---

## 1. 기술 선택

| 항목 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | **React Native + Expo** | 단일 코드로 iOS·Android, OTA 업데이트, 빠른 개발 |
| 언어 | TypeScript | 백엔드 스키마와 타입 공유 |
| 내비게이션 | React Navigation (Bottom Tabs + Stack) | 표준 |
| 상태/통신 | TanStack Query + fetch | 서버 상태 캐싱·재검증 |
| 오디오 녹음 | `expo-av` / `expo-audio` | 마이크 녹음·재생 |
| 인앱 결제 | **RevenueCat** 또는 스토어 IAP | 아래 4장 참고 |
| 보안 저장 | `expo-secure-store` | 토큰 |

> 웹(React)과 **API 클라이언트·타입·도메인 로직**을 공유하고, UI 레이어만 RN으로 구현한다. 모노레포(예: pnpm workspaces)로 `packages/api`, `apps/web`, `apps/mobile` 분리 권장.

## 2. 화면 구조

```
RootStack
├─ Onboarding (프로필: 이메일/모국어/레벨)
└─ MainTabs (하단 탭)
   ├─ Store        목소리 카탈로그 · 미리듣기 · 구매
   ├─ MyVoices     보유 목소리
   ├─ Register     음성 녹음/업로드 + 동의 게이트
   └─ Chat         대화 (텍스트/음성 입력, 복제 목소리 재생)
   └─ Settings     계정 · 데이터 삭제 · 약관
```

### 화면별 핵심

- **Store**: 억양·성별 필터, 카드 목록, 미리듣기 버튼, 가격, [구매]/[보유중] 상태. 구매는 인앱결제 시트.
- **Register**:
  1. 마이크로 1~5분 녹음 또는 파일 선택.
  2. **동의 화면** — 5개 항목 체크(모두 필수), 미체크 시 등록 버튼 비활성. 동의 문구는 웹과 동일.
  3. 업로드 → 처리중 → 완료. (선택) 판매 공개 설정.
- **Chat**: 상단에 목소리 선택 드롭다운, 메시지 버블, 봇 응답에 오디오 플레이어 + 교정 피드백. 하단 입력바에 텍스트 + 🎙(녹음→`/chat/transcribe`) 버튼.
- **Settings**: "내 목소리/데이터 완전 삭제"(DELETE /voices/{id}), 약관·개인정보 처리방침, 로그아웃.

## 3. 권한 & 플랫폼 설정

- **마이크 권한**: iOS `NSMicrophoneUsageDescription`, Android `RECORD_AUDIO`. 최초 녹음 시 권한 요청, 거부 시 안내.
- **백그라운드 오디오 재생**: iOS `UIBackgroundModes: audio` (필요 시).
- **네트워크**: HTTPS 강제(iOS ATS). 로컬 개발은 예외 설정.

---

## 4. 인앱 결제 (중요)

앱스토어·플레이스토어는 **디지털 콘텐츠 구매에 자체 IAP 사용을 강제**한다. 따라서 모바일에서 목소리 구매는 웹의 Stripe 플로우를 그대로 쓸 수 없고 다음을 따른다.

- **iOS/Android**: StoreKit / Google Play Billing (또는 이를 통합한 **RevenueCat**) 사용. 상품을 스토어 콘솔에 등록하고, 구매 영수증을 백엔드가 검증 → `Entitlement` 부여.
- **백엔드 연동**: 기존 `Order/Entitlement` 모델을 재사용하되, `provider="apple"|"google"`, `provider_ref=영수증/트랜잭션ID`로 확정. 서버영수증검증 후 보유권 부여.
- **웹↔앱 보유권 공유**: 동일 계정이면 보유권이 공유되므로, 웹에서 산 목소리를 앱에서도 사용 가능(반대도 동일). 단, 결제 경로는 플랫폼 규정상 분리.
- **수수료 주의**: 스토어 수수료(약 15~30%)를 가격·정산 모델에 반영.

## 5. 오디오 처리 흐름 (모바일)

```
[음성 입력] expo-av 녹음(m4a/wav)
   → 업로드 POST /chat/transcribe (Whisper) → 텍스트
   → POST /chat → reply_text + feedback + audio_url
   → expo-av로 audio_url 재생 (복제 목소리)
```

지연시간 개선: 백엔드 스트리밍 TTS 도입 시, RN에서 `expo-av` 스트리밍 재생 또는 청크 재생으로 첫 음절 빠르게.

---

## 6. 동의·악용방지 (모바일에서도 동일 적용)

- 등록 화면의 동의 게이트는 서버가 최종 검증(5개 항목 누락 시 403). 클라이언트 UX는 보조.
- 대화 입력은 서버 모더레이션을 그대로 통과 → 차단 시 경고 버블.
- Settings의 완전 삭제로 **삭제권** 보장(생체정보 규제 대응).
- 합성 오디오는 서버가 워터마크 메타데이터 부착.

---

## 7. 배포

- **Expo EAS Build**로 iOS/Android 바이너리 생성, **EAS Submit**으로 스토어 제출.
- OTA(Expo Updates)로 JS 변경 즉시 배포(네이티브 변경은 재심사).
- 개인정보·생체정보 처리 고지를 스토어 심사 자료(App Privacy)에 정확히 기재.

---

## 8. 구현 우선순위 (모바일)

1. Onboarding + Chat(텍스트) + 보유 목소리 재생 — 핵심 가치 검증.
2. Register(녹음 + 동의) — 입력 파이프라인.
3. Store + IAP(RevenueCat) — 수익화.
4. 음성 입력(STT) + 스트리밍 재생 — 실시간 대화 경험.
