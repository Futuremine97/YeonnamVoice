# English For Us — 음성복제 기반 영어회화 학습 웹서비스 기술 설계 문서

> **버전** 1.0 · **작성일** 2026-06-13
> **한 줄 요약** 원어민/비원어민 영어 사용자의 1~5분 음성만으로 그 사람의 목소리를 복제하고, 그 목소리로 한국·일본 학습자와 실시간 영어 대화를 나누는 웹서비스.

---

## 1. 제품 개요

### 1.1 핵심 사용자 시나리오

1. 사용자가 듣고 싶은 영어 화자(예: 좋아하는 강사, 친구, 본인 롤모델)의 음성 1~5분을 업로드한다.
2. 시스템이 그 음성에서 **목소리 지문(voice profile)** 을 추출·저장한다 (수 초~수십 초 소요).
3. 학습자(한국인·일본인)는 채팅창에 텍스트를 입력하거나 마이크로 영어를 말한다.
4. AI가 영어로 대화를 이어가고, **복제된 목소리로** 답변을 음성 재생한다.
5. AI는 동시에 학습자의 발음·문법을 교정하고, 필요하면 한국어/일본어로 설명을 덧붙인다.

### 1.2 두 개의 분리된 음성 처리

이 서비스에는 성격이 완전히 다른 두 개의 음성 파이프라인이 있다. 설계 내내 이 둘을 혼동하지 않는 것이 중요하다.

| 구분 | 입력 | 처리 | 목적 |
|------|------|------|------|
| **A. 화자 목소리 복제** | 영어 화자의 1~5분 샘플 | 오픈소스 로컬 TTS (Chatterbox 등) | AI 답변을 그 목소리로 출력 |
| **B. 학습자 발화 인식** | 학습자가 말하는 영어 | 클라우드 STT (Whisper API) | 학습자 말을 텍스트로 변환 → 대화·교정 |

A는 "출력용 목소리"를 만드는 것이고, B는 "학습자 입력"을 받아들이는 것이다.

## 2. 기술 스택 결정 요약

사전 협의에 따라 다음과 같이 확정한다.

| 영역 | 선택 | 비고 |
|------|------|------|
| 음성 복제(TTS) | **오픈소스 로컬** | 무료·프라이버시·자체 호스팅 |
| 대화 엔진(LLM) | **클라우드 API** (Claude / OpenAI) | 영어 교정·다국어 설명 품질 |
| 음성 인식(STT) | **클라우드 API** (Whisper 계열) | 억양 있는 학습자 발화 정확도 |

## 3. 음성 복제 모델 선정 (핵심)

2026년 6월 기준 오픈소스 few-shot/zero-shot 음성복제 모델을 조사한 결과는 다음과 같다.

| 모델 | 라이선스 | 필요 샘플 | 강점 | 약점 |
|------|----------|-----------|------|------|
| **Chatterbox / Chatterbox-Turbo** (Resemble AI) | **MIT (상업적 사용 가능)** | 약 5초 | 스트리밍 first-packet ~150ms, GPU에서 실시간 대비 6배, 블라인드 테스트에서 ElevenLabs보다 선호(65% vs 24%), Turbo는 350M 파라미터로 VRAM 부담 적음 | 상대적으로 신생, 한국어 발화는 목표 아님(영어 출력만 필요하므로 무관) |
| **F5-TTS** | 비상업/연구 성격(저장소·체크포인트별 확인 필요) | 약 3초 | flow-matching, 빠른 추론, 활발한 커뮤니티 | 라이선스 모호, 프로덕션 사용 시 확인 필요 |
| **XTTS-v2** (Coqui) | **Coqui Public Model License — 비상업** | 약 6초, 17개 언어 | 다국어, 안정적 | **상업 서비스에는 별도 라이선스 필요**, 4~6GB VRAM |
| Fish Speech / CosyVoice2 / IndexTTS-2 | 모델별 상이 | 수 초 | 최신·고품질 후보 | 라이선스·운영 난이도 개별 검토 |

### 3.1 권장안

**1순위: Chatterbox-Turbo.** 이유:

- **MIT 라이선스** — 향후 유료화·상업 서비스로 확장해도 법적 제약이 없다. (XTTS-v2의 가장 큰 함정은 비상업 라이선스다.)
- **낮은 지연시간** — 대화형 서비스는 응답이 1~2초만 늦어도 체감이 크게 나빠진다. Chatterbox는 스트리밍 출력을 지원해 첫 음절을 빠르게 내보낸다.
- **작은 VRAM** — 350M급으로 중급 GPU(또는 일부 고성능 맥)에서도 구동 가능.

**대안: F5-TTS** (품질 비교용), **XTTS-v2** (다국어가 필요하고 비상업/개인용으로 한정할 경우).

> 설계 원칙: TTS 엔진을 **인터페이스 뒤로 추상화**(`TTSEngine` 추상 클래스)하여 Chatterbox ↔ F5-TTS ↔ XTTS를 코드 변경 최소화로 교체할 수 있게 한다.

### 3.2 "1~5분 샘플"을 잘 쓰는 법

대부분의 모델은 단 몇 초의 참조 음성만으로 복제가 가능하다. 그렇다면 1~5분을 받는 이유는 **품질을 높이기 위해서**다. 긴 샘플은 다음 전처리에 활용한다.

1. **VAD(음성구간검출)** 로 무음·잡음 구간 제거.
2. **잡음제거/정규화** (예: `noisereduce`, `pyloudnorm`).
3. 깨끗하고 운율이 풍부한 **베스트 구간 자동 선별**(SNR·길이 기준 상위 N개).
4. 여러 참조 구간으로 **스피커 임베딩을 평균화**해 안정적인 목소리 지문 생성.
5. (선택) 5분 이상 충분한 데이터가 모이면 **경량 파인튜닝**으로 유사도 추가 향상.

## 4. 대화 엔진 & STT

### 4.1 대화(LLM)

- **모델**: Claude (Sonnet/Opus) 또는 OpenAI GPT-4o 계열.
- **역할**:
  - 학습자 레벨에 맞춘 영어 대화 상대(롤플레이, 자유 대화, 주제별 회화).
  - 발화 후 **문법·표현·발음 피드백** 생성.
  - 한국어/일본어로 핵심 설명 병기(학습자 모국어 감지 또는 설정값 사용).
- **구현 포인트**: 시스템 프롬프트에 (a) 학습자 모국어·레벨, (b) 교정 방식, (c) 대화 시나리오를 주입. 대화 맥락은 세션별로 유지.

### 4.2 음성 인식(STT)

| 옵션 | 가격(2026) | 특징 |
|------|-----------|------|
| OpenAI Whisper API | $0.006/분 | 파일 기반, 억양 견고, 실시간 스트리밍 아님 |
| GPT-4o-mini-transcribe | $0.003/분 | 더 저렴, 빠름 |
| Deepgram / AssemblyAI 스트리밍 | 별도 | **실시간 스트리밍** 가능 — 라이브 대화에 유리 |

- **MVP**: Whisper API(또는 GPT-4o-mini-transcribe)로 "녹음 후 전송" 방식. 구현 단순.
- **고도화**: 실시간 대화 느낌을 살리려면 Deepgram/AssemblyAI 스트리밍 STT로 전환.

## 5. 시스템 아키텍처

```
┌──────────────────────────────────────────────────────────────┐
│                        Frontend (React)                        │
│  · 음성 업로드(1~5분)   · 채팅 UI   · 마이크 녹음   · 오디오 재생  │
└───────────────┬───────────────────────────────┬───────────────┘
                │ HTTPS / WebSocket             │
┌───────────────▼───────────────────────────────▼───────────────┐
│                    Backend API (FastAPI · Python)              │
│   ┌──────────────┐  ┌──────────────┐  ┌────────────────────┐   │
│   │ 음성 등록     │  │ 대화 오케스   │  │ 사용자/세션 관리    │   │
│   │ (전처리·지문) │  │ 트레이션      │  │                    │   │
│   └──────┬───────┘  └──────┬───────┘  └────────────────────┘   │
└──────────┼──────────────────┼──────────────────────────────────┘
           │                  │
   ┌───────▼────────┐   ┌─────▼──────────────────────────────────┐
   │ TTS 엔진(로컬)  │   │  외부 클라우드 API                       │
   │ Chatterbox     │   │  · STT: Whisper API                     │
   │ (GPU)          │   │  · LLM: Claude / OpenAI                  │
   └───────┬────────┘   └──────────────────────────────────────────┘
           │
   ┌───────▼────────────────────────────────────────────────────┐
   │  저장소:  Postgres(메타)  ·  객체스토리지(음성/지문)  ·  Redis │
   └─────────────────────────────────────────────────────────────┘
```

### 5.1 데이터 흐름 — ① 목소리 등록

```
업로드(1~5분) → VAD·잡음제거·정규화 → 베스트 구간 선별 →
스피커 임베딩(목소리 지문) 생성 → 객체스토리지 저장 → voice_id 발급
```

### 5.2 데이터 흐름 — ② 대화 (텍스트 입력)

```
학습자 텍스트 → LLM(영어 답변 + 교정/모국어 설명) →
답변 텍스트 → TTS(voice_id로 합성) → 오디오 스트림 → 재생
```

### 5.3 데이터 흐름 — ③ 대화 (음성 입력)

```
마이크 녹음 → STT(Whisper) → 텍스트 → (이후 ②와 동일) →
복제 목소리로 음성 답변 + 발음 피드백
```

## 6. 기술 스택 상세

| 레이어 | 기술 | 선정 이유 |
|--------|------|-----------|
| 프론트엔드 | React + TypeScript, Web Audio API / MediaRecorder | 녹음·재생·스트리밍 UI |
| 백엔드 | **Python + FastAPI** | TTS 모델(파이썬 생태계)과 동일 런타임, async/WebSocket 지원 |
| TTS 런타임 | PyTorch + Chatterbox | GPU 추론 |
| STT/LLM | OpenAI / Anthropic SDK | 클라우드 호출 |
| DB | PostgreSQL | 사용자·세션·voice 메타데이터 |
| 캐시/큐 | Redis | 세션, 작업 큐 |
| 객체 스토리지 | S3 호환(예: MinIO 로컬, 또는 클라우드 S3) | 음성 원본·지문·생성 오디오 |
| 배포 | Docker + (GPU 노드) | 재현성, TTS는 GPU 컨테이너 분리 |

> 프론트는 Next.js로 통합해도 무방하나, **TTS가 파이썬**이므로 백엔드는 FastAPI를 권장한다. (Node 백엔드라면 TTS를 별도 파이썬 마이크로서비스로 분리.)

## 7. 하드웨어 요구사항

- **TTS GPU**: Chatterbox-Turbo 기준 VRAM 약 4GB 이상 권장(XTTS는 4~6GB). 실시간성을 위해 NVIDIA GPU 권장.
- **로컬 개발(맥)**: Apple Silicon에서 MPS 백엔드로 구동 가능하나, 지연시간은 전용 GPU보다 느릴 수 있음. 초기 프로토타입엔 충분.
- **클라우드 GPU 옵션**: 사용자 트래픽이 늘면 GPU 인스턴스(예: T4/L4급)로 TTS 서비스만 분리 배포.

| 항목 | 단가 | 비고 |
|------|------|------|
| 음성 복제(TTS) | **$0 (로컬)** | 전기·GPU 감가만 발생 |

## 9. 단계별 구현 로드맵

### Phase 0 — 검증 (1~2주)
- Chatterbox로 샘플 음성 복제 품질 단독 테스트 (CLI 수준).
- Whisper API + LLM로 텍스트 대화 루프 단독 테스트.

### Phase 1 — MVP (3~4주)
- 음성 업로드 → 지문 생성 → voice_id.
- 텍스트 채팅 → LLM 답변 → 복제 목소리 재생.
- 단순 웹 UI(녹음·재생·채팅).

### Phase 2 — 음성 대화 (2~3주)
- 마이크 녹음 → Whisper STT → 대화 루프.
- 발음·문법 피드백, 모국어(한/일) 설명.

### Phase 3 — 품질·실시간 (2~3주)
- 베스트 구간 선별·잡음제거 전처리 고도화.
- 스트리밍 STT/TTS로 지연시간 최소화.
- 레벨별 시나리오·롤플레이 모드.

### Phase 4 — 운영화 (필요 시)
- 회원/인증, 사용량 제한, GPU 오토스케일, 모니터링.

## 10. 리스크 & 법적·윤리 고려사항 (중요)

음성 복제 서비스는 기술보다 **동의·악용 방지**가 더 큰 리스크다.

1. **목소리 주인의 동의**: 타인의 목소리를 복제하려면 본인 동의가 필수다. 업로드 시 "본인 또는 동의받은 음성"임을 확인하는 체크·약관을 두고, 가능하면 동의 증빙을 요구한다.
2. **딥페이크/사칭 방지**: 합성 음성에 (가능하면) 워터마크·식별자를 남기고, 유명인·정치인 사칭 등 악용을 약관으로 금지.
3. **미성년자 보호**: 미성년자 목소리 업로드·대상화 차단.
4. **개인정보**: 음성은 민감 생체정보일 수 있다. 저장·삭제 정책, 암호화, 보관기간을 명시. 사용자가 언제든 삭제 요청 가능하게.
5. **라이선스 준수**: 채택 모델의 라이선스를 프로덕션 전에 재확인(특히 XTTS-v2는 비상업). 본 설계의 1순위 Chatterbox는 MIT라 안전.
6. **품질 한계**: 5초~5분 샘플로는 감정·억양이 완벽히 재현되지 않을 수 있음 — 학습용으로는 충분하나 기대치 관리 필요.
